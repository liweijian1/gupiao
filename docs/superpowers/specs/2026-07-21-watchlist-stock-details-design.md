# 自选股票行情详情设计

## 目标

让“自选”范围直接从服务端读取当前登录用户保存的股票及其可渲染行情详情。展示不能再依赖当前市场快照中是否已包含某个自选代码。

## 范围

- 保持现有邮箱账户、Cookie 会话和 SQLite 自选记录不变。
- 新增 `GET /api/watchlist/stocks`，仅对已登录用户可用。
- 左侧“自选”页签使用该接口的数据渲染排名列表。
- 不新增交易、价格提醒、分类文件夹或自选共享功能。

## 接口

`GET /api/watchlist/stocks` 返回：

```json
{
  "tickers": ["002197", "600519"],
  "stocks": [{ "ticker": "002197", "name": "示例股票" }],
  "unavailable_tickers": []
}
```

- 未登录返回现有 `401 authentication_required`。
- `tickers` 保持数据库 `created_at` 的保存顺序。
- `stocks` 与已成功解析的 `tickers` 顺序一致。
- 代码未能获取行情时保留在 `unavailable_tickers`，不会从数据库删除。

## 后端数据流

1. `require_current_user` 获取会话对应用户。
2. `AuthStore.list_watchlist_tickers` 读取该用户的保存代码。
3. 先读取当前股票快照，以代码建立索引。
4. 快照没有的代码调用现有单股 `stock_quote` 查询；查询失败或没有股票结果时记录到 `unavailable_tickers`。
5. 按自选保存顺序组装结果。接口不写入或重排自选数据库记录。

## 前端数据流

1. `useWatchlist` 在登录态恢复后加载代码列表，并提供独立的详情刷新动作。
2. 用户打开“自选”范围时请求 `GET /api/watchlist/stocks`；已有列表在刷新期间保留。
3. `EquityDiscoveryPanel` 对自选范围只使用服务端返回的 `stocks`，不再与当前市场筛选结果取交集。
4. 无可解析行情时显示已有的空状态；部分不可用时保留已获取行并给出简短提示。

## 验收与测试

- 后端测试覆盖用户隔离、保存顺序、快照命中、缺失代码的单股补查和不可用代码回传。
- 前端测试覆盖自选范围调用详情接口并用返回行渲染，不依赖当前市场快照。
- 浏览器验证：保存不在当前排名快照中的代码后，重新打开“自选”仍显示该股票；登出后接口与范围均不泄露私有列表。
